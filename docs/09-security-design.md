# PharmacyCRM — Security Design

**Статус документа:** Draft  
**Версия:** 0.1  
**Дата:** 2026-07-17

## 1. Назначение документа

Документ определяет обязательную модель безопасности PharmacyCRM: защищаемые активы, границы доверия, модель угроз, аутентификацию, управление сессиями, авторизацию, защиту HTTP API и браузерного клиента, криптографию, управление секретами, аудит, журналирование, защиту данных, безопасную разработку и требования к реагированию на инциденты.

Security Design является нормативным для backend, frontend, инфраструктуры, тестов и эксплуатационных процедур. Реализация функции считается незавершённой, если её критические security controls не реализованы и не покрыты тестами.

Документ не заменяет:

- `01-product-vision.md` — продуктовые цели и границы MVP;
- `02-srs.md` — обязательное внешнее поведение системы;
- `03-system-context.md` — системную границу, акторов и trust boundaries;
- `04-architecture.md` и `04-01-backend-architecture.md` — архитектурные слои, модули и dependency rules;
- `05-api-design.md` — HTTP-контракты;
- `06-database-design.md` — таблицы, ограничения и модель хранения;
- `07-domain-model.md` — агрегаты, инварианты и транзакционные границы;
- `08-project-structure.md` — физическую структуру backend и frontend;
- ADR — принятые архитектурно значимые решения.

При противоречии применяется порядок приоритетов, установленный в SRS. Security Design не должен незаметно расширять полномочия ролей или изменять бизнес-инварианты.

## 2. Цели безопасности

PharmacyCRM должна обеспечивать:

1. конфиденциальность учётных данных, токенов, персональных и внутренних операционных данных;
2. целостность складских, торговых, ценовых и административных операций;
3. невозможность обхода ограничений роли и назначения аптекаря конкретной аптеке;
4. невозможность незаметного изменения проведённых документов, складских движений и аудита;
5. контролируемое прекращение доступа заблокированного, архивированного или скомпрометированного пользователя;
6. устойчивость критических команд к повторной отправке, гонкам и подмене payload;
7. трассируемость действий пользователя, администратора и системного процесса;
8. минимизацию последствий компрометации отдельного токена, браузера, аккаунта или инфраструктурного компонента;
9. безопасное хранение и ротацию секретов;
10. обнаружение злоупотреблений и достаточные данные для расследования инцидентов;
11. безопасные значения по умолчанию и отказ в доступе при неопределённости;
12. проверяемость security controls автоматическими тестами.

## 3. Принципы безопасности

### 3.1 Недоверенный клиент

Браузер, frontend-код, мобильный клиент, HTTP-запрос и любые переданные клиентом идентификаторы являются недоверенными.

Backend обязан самостоятельно определять и проверять:

- идентичность пользователя;
- состояние пользователя и сессии;
- роль;
- назначение аптекаря конкретной аптеке;
- право на конкретное действие над конкретным ресурсом;
- итоговую цену и денежную сумму;
- доступный остаток;
- допустимые лоты и FEFO-распределение;
- серверное время операции;
- допустимый переход состояния;
- соответствие idempotency key исходному смысловому payload.

Скрытие кнопки или маршрута во frontend не является авторизацией.

### 3.2 Deny by default

Неизвестная роль, неизвестное состояние, отсутствующее назначение, неполный security context, ошибка чтения пользователя или сессии, ошибка проверки политики и неоднозначный результат должны приводить к отказу в доступе.

Security middleware и policy layer не должны использовать fail-open поведение.

### 3.3 Least privilege

Каждый пользователь, процесс, токен, сервисный аккаунт и роль БД получают только минимальные полномочия, необходимые для конкретной функции.

Доступ не должен предоставляться «на будущее». Новые разрешения добавляются явно и сопровождаются обновлением документации и тестов.

### 3.4 Defense in depth

Критический инвариант защищается несколькими уровнями:

- HTTP validation;
- authentication middleware;
- application authorization policy;
- domain invariants;
- транзакционные проверки;
- ограничения и блокировки PostgreSQL;
- аудит и мониторинг.

Ни один отдельный уровень не считается достаточным.

### 3.5 Complete mediation

Каждая защищённая операция должна заново проходить необходимую проверку доступа. Нельзя полагаться на то, что пользователь ранее открыл страницу, получил список или успешно выполнил связанную команду.

### 3.6 Separation of duties

Технический оператор инфраструктуры не является бизнес-администратором. Доступ к секретам, резервным копиям и БД не должен автоматически давать штатное право менять бизнес-данные.

Административная роль приложения не даёт права незаметно переписывать историю операций.

### 3.7 Security by construction

Безопасность должна выражаться в архитектурных контрактах, типах, policy-функциях, транзакционных сценариях и ограничениях БД, а не зависеть от дисциплины отдельных handler-ов.

## 4. Защищаемые активы

### 4.1 Активы критической важности

- пароли и password hashes;
- access tokens, refresh tokens и session records;
- роли и назначения пользователей аптекам;
- проведённые продажи, возвраты, поступления, списания и корректировки;
- складские движения и остатки по лотам;
- закупочные и продажные цены;
- idempotency records;
- audit events;
- секреты подписи токенов, ключи шифрования и credentials внешних систем;
- резервные копии PostgreSQL;
- administrative endpoints;
- pipeline сборки и публикации артефактов.

### 4.2 Чувствительные внутренние данные

- точные служебные остатки;
- закупочные цены;
- внутренние причины списания и корректировки;
- данные поставок и поставщиков;
- история действий сотрудников;
- технические логи, IP-адреса и user-agent;
- импортируемые файлы до модерации;
- сведения об инцидентах.

### 4.3 Публичные данные

Публичными могут быть только поля, явно предусмотренные публичным API: данные опубликованного каталога, разрешённые сведения об аптеке, публичная цена, агрегированный статус наличия и время актуальности.

Точная величина остатка, идентификаторы внутренних лотов, закупочная цена, внутренние документы и audit trail не должны попадать в публичный контур.

## 5. Границы доверия

Основные trust boundaries:

1. публичный интернет → reverse proxy / ingress;
2. браузер → frontend application;
3. frontend → backend HTTP API;
4. backend delivery → application layer;
5. application → infrastructure adapters;
6. backend → PostgreSQL;
7. backend → файловое хранилище или внешние интеграции;
8. CI/CD → registry и production environment;
9. оператор → infrastructure management plane;
10. backup storage → restore process.

Пересечение каждой границы требует явной валидации, аутентификации канала или ограничения полномочий в зависимости от типа потока.

## 6. Модель угроз

### 6.1 Внешний атакующий

Возможные цели:

- подбор пароля;
- credential stuffing;
- кража или повторное использование токена;
- эксплуатация injection, XSS, CSRF, SSRF, path traversal;
- перечисление пользователей и внутренних идентификаторов;
- scraping публичного поиска;
- denial of service;
- загрузка вредоносного файла;
- эксплуатация ошибочной конфигурации CORS или proxy headers.

### 6.2 Злоумышленник с учётной записью CLIENT

Возможные цели:

- доступ к внутреннему API;
- повышение роли через mass assignment;
- чтение служебных остатков и цен;
- горизонтальный доступ к чужим ресурсам;
- злоупотребление публичными и персональными endpoint-ами.

### 6.3 Злоумышленник или скомпрометированный PHARMACIST

Возможные цели:

- работа с чужой аптекой через подмену `pharmacy_id`;
- продажа запрещённого, просроченного или карантинного лота;
- обход FEFO;
- изменение итоговой суммы;
- повторное проведение команды;
- создание фиктивных корректировок;
- удаление следов;
- извлечение закупочных и внутренних данных за пределами полномочий.

### 6.4 Злоумышленник или скомпрометированный ADMIN

Возможные цели:

- назначение избыточных ролей;
- создание скрытого администратора;
- изменение истории операций;
- массовый экспорт чувствительных данных;
- отключение аудита;
- блокировка легитимных пользователей;
- компрометация каталога и публичных данных.

ADMIN считается высокорисковой ролью. Её действия требуют усиленного аудита и, для особенно опасных сценариев, повторной аутентификации или дополнительного подтверждения.

### 6.5 Инсайдер с инфраструктурным доступом

Возможные цели:

- чтение секретов и backup;
- прямое изменение БД;
- подмена артефакта;
- отключение логирования;
- извлечение данных из логов;
- создание неучтённого доступа.

### 6.6 Компрометированная зависимость или pipeline

Возможные цели:

- внедрение вредоносного кода;
- кража CI secrets;
- публикация подменённого контейнера;
- выполнение произвольного кода через dependency lifecycle script;
- утечка credentials через build logs.

## 7. Идентичность пользователя

### 7.1 Идентификатор входа

Идентификатор входа должен быть уникальным в пределах принятой политики нормализации. До утверждения повторного использования ранее занятого идентификатора система не должна назначать его другому активному пользователю.

Сравнение выполняется по нормализованному значению. Исходное отображаемое значение может храниться отдельно.

API не должен раскрывать, существует ли пользователь, через различающиеся тексты или заметно различающееся поведение login/reset flow.

### 7.2 Состояния пользователя

Минимально различаются состояния:

- active;
- blocked;
- archived.

Только active user может создавать новую сессию и выполнять защищённые операции.

Blocked user теряет возможность аутентификации, а его активные сессии должны быть отозваны или стать недействительными в пределах установленного security SLA.

Archived user не должен восстанавливаться неявно через обычное редактирование профиля.

### 7.3 Системный инициатор

Фоновые задачи и maintenance operations используют отдельный тип actor context. Они не должны маскироваться под обычного пользователя или фиктивный ADMIN.

Audit event должен различать human actor, system actor и, при появлении интеграций, service actor.

## 8. Пароли

### 8.1 Хранение

Пароли хранятся только как стойкий password hash. Обратимое шифрование паролей запрещено.

Предпочтительный алгоритм — Argon2id с параметрами, подобранными нагрузочным тестированием целевой среды. Допустимо использование bcrypt только при отдельном ADR или для совместимости с существующими hashes.

Password record должен хранить алгоритм и параметры так, чтобы поддерживать постепенный rehash после успешного входа.

### 8.2 Требования к паролю

Для PHARMACIST и CLIENT пароль должен иметь длину не менее 12 символов. Для ADMIN — не менее 14 символов.

Максимальная принимаемая длина должна быть ограничена, но не должна обрезаться молча. Рекомендуемый предел — 128 Unicode code points после установленной политики нормализации.

Система должна:

- разрешать password manager friendly passwords и passphrases;
- не требовать искусственной периодической смены без признаков компрометации;
- запрещать широко скомпрометированные и очевидно слабые пароли;
- не использовать секретные вопросы;
- не логировать пароль или его производные;
- не передавать пароль в URL, query string или analytics.

### 8.3 Создание и сброс

Самостоятельная регистрация PHARMACIST и ADMIN запрещена.

Временный пароль должен быть одноразовым и требовать смены при первом входе либо заменяться безопасной activation flow.

Reset token:

- генерируется криптографически стойким генератором;
- хранится в БД только как hash;
- имеет короткий TTL;
- одноразовый;
- привязан к конкретному пользователю и назначению;
- инвалидируется после использования, смены пароля, блокировки или выпуска нового reset token.

После смены или сброса пароля все существующие refresh sessions пользователя должны быть отозваны, кроме явно выбранной текущей сессии при утверждённой политике.

## 9. Аутентификация и сессии

### 9.1 Модель токенов

Для web-клиента используется короткоживущий access token и серверно контролируемая refresh session.

Access token подтверждает ранее выполненную аутентификацию, но не является единственным источником полномочий для критических операций.

Access token должен содержать минимальный набор claims:

- `sub` — user ID;
- `sid` — session ID;
- `iat`;
- `exp`;
- `iss`;
- `aud`;
- уникальный token ID при необходимости denylist или расследования.

Роль и назначение могут присутствовать как оптимизационная подсказка, но backend обязан повторно проверять актуальное состояние пользователя и назначение для критических и административных операций.

Чувствительные персональные данные и список полномочий не должны помещаться в token payload.

### 9.2 Сроки жизни

Рекомендуемые начальные значения:

- access token TTL: 5–15 минут;
- refresh session idle TTL: до 7 дней;
- абсолютный TTL refresh session: до 30 дней для CLIENT и PHARMACIST;
- абсолютный TTL ADMIN session: не более 12 часов;
- период повторной аутентификации ADMIN для опасных операций: не более 15 минут.

Конкретные значения должны быть конфигурируемыми, проверяться при startup и фиксироваться в deployment configuration.

### 9.3 Хранение токенов в браузере

Refresh token должен передаваться только в cookie со свойствами:

- `HttpOnly`;
- `Secure` в production;
- `SameSite=Lax` или `Strict` в зависимости от утверждённого frontend deployment;
- узкий `Path` только для refresh/logout endpoint;
- без широкого `Domain`, если он не требуется.

Хранение refresh token в `localStorage`, `sessionStorage`, IndexedDB или доступном JavaScript state запрещено.

Access token предпочтительно хранится только в памяти приложения. Если выбран cookie-based access token, обязательна полноценная CSRF-защита всех state-changing requests.

### 9.4 Refresh token rotation

Каждая успешная refresh operation должна:

1. проверить активность пользователя и session record;
2. проверить hash текущего refresh token;
3. атомарно пометить текущий token generation использованным;
4. выпустить новую token generation;
5. сохранить новый hash;
6. вернуть новый refresh token;
7. зарегистрировать security event.

Повторное использование ранее ротированного refresh token считается признаком кражи. В этом случае вся session family должна быть отозвана, событие зафиксировано в audit/security log, а пользователь может быть принудительно разлогинен на связанных устройствах согласно политике.

Refresh token должен иметь не менее 256 бит энтропии и генерироваться CSPRNG.

### 9.5 Session record

Server-side session должна содержать как минимум:

- session ID;
- user ID;
- refresh token hash;
- token family / generation;
- created at;
- last used at;
- idle expires at;
- absolute expires at;
- revoked at и revoke reason;
- optional IP prefix и user-agent fingerprint для расследования, но не как единственный фактор авторизации.

Raw refresh token не хранится.

### 9.6 Logout и отзыв

Logout должен атомарно отзывать server-side session и очищать refresh cookie.

Должны поддерживаться:

- logout текущей сессии;
- logout всех сессий пользователя;
- административный отзыв всех сессий;
- автоматический отзыв при блокировке, архивировании, смене пароля и обнаружении reuse.

Операция logout должна быть идемпотентной.

### 9.7 Многофакторная аутентификация

Для production ADMIN MFA является обязательной целью до допуска удалённого административного доступа. Предпочтительны TOTP или WebAuthn.

Для PHARMACIST MFA может быть включена политикой аптеки или при повышенном риске.

SMS не должен быть единственным фактором для ADMIN, если доступны более устойчивые методы.

### 9.8 Защита login endpoint

Login flow должен иметь:

- rate limiting по комбинации account identifier, IP и global pressure;
- прогрессивную задержку или временный cooldown;
- одинаковый внешний ответ для неизвестного пользователя и неверного пароля;
- метрику неуспешных входов;
- security event при подозрительной активности;
- защиту от distributed credential stuffing на уровне ingress/WAF при наличии.

Жёсткая постоянная блокировка аккаунта только по числу неудач запрещена, так как позволяет атакующему блокировать пользователей. Допустим временный server-side throttle.

## 10. Авторизация

### 10.1 Общая модель

Авторизация сочетает:

- RBAC — разрешения, зависящие от роли;
- resource scope — принадлежность ресурса аптеке;
- assignment check — активное назначение PHARMACIST аптеке;
- account/session state;
- business state — допустимость действия в текущем состоянии агрегата;
- contextual constraints — повторная аутентификация, idempotency, server time и другие условия.

Проверка только роли недостаточна.

### 10.2 Роли

Минимальные роли:

- `CLIENT`;
- `PHARMACIST`;
- `ADMIN`.

Неизвестная роль не получает никаких полномочий.

Назначение роли выполняется только через отдельный административный use case. Role field не должен приниматься через общий profile update DTO.

### 10.3 Назначение аптекаря

Для каждого pharmacy-scoped use case backend обязан:

1. получить actor ID из проверенного authentication context;
2. загрузить актуальное состояние пользователя;
3. определить target pharmacy из ресурса или команды;
4. проверить активное назначение PHARMACIST этой аптеке;
5. проверить состояние аптеки;
6. только после этого выполнять чтение или изменение.

Переданный клиентом `pharmacy_id` не является доказательством доступа.

Для команды над существующим ресурсом pharmacy scope должен по возможности выводиться из загруженного ресурса, а не только из route parameter.

### 10.4 Application policy layer

Авторизационные правила должны быть централизованы в явных policy-компонентах или application services. Запрещено копировать разрозненные role checks по handler-ам.

Policy API должен возвращать доменную/application ошибку и работать без зависимости от Gin или HTTP status code.

Handler отвечает только за извлечение authentication context и вызов use case.

### 10.5 Повторная проверка в транзакции

Для критических mutation use case состояние пользователя, роль и назначение должны быть повторно проверены внутри той же транзакции, в которой выполняется бизнес-изменение.

К таким операциям относятся:

- проведение продажи;
- возврат;
- поступление;
- списание и корректировка;
- изменение цен;
- изменение ролей и назначений;
- блокировка пользователя или аптеки;
- публикация каталога;
- административные исправления;
- операции с секретами и сессиями.

Это предотвращает stale authorization, когда право было отозвано между предварительной проверкой и commit.

### 10.6 Административные операции

Для ADMIN должны применяться дополнительные требования:

- повторная аутентификация для изменения ролей, блокировок, экспорта, security settings и других опасных операций;
- обязательный reason/comment для корректирующих действий;
- отдельный audit event до возврата успешного ответа;
- защита от self-demotion/self-block или иных действий, способных оставить систему без управляемого администратора, согласно отдельной политике;
- запрет универсального endpoint-а произвольного обновления полей;
- явные DTO allowlist;
- возможность принципа four-eyes для будущих особо критических действий.

### 10.7 Проверка доступа к чтению

Read endpoint также обязан проверять scope. Нельзя считать чтение безопасным только потому, что оно не меняет данные.

Особенно защищаются:

- закупочные цены;
- точные остатки и лоты;
- поставки;
- audit events;
- user/session lists;
- импортируемые файлы;
- причины корректировок;
- административные отчёты.

## 11. Защита HTTP API

### 11.1 TLS

Production API доступен только по HTTPS. HTTP должен перенаправляться на HTTPS либо быть недоступен на внешней границе.

TLS завершается на контролируемом ingress/reverse proxy. Соединение proxy → backend также должно быть защищено или находиться в изолированной доверенной сети с документированной моделью риска.

### 11.2 Proxy headers

Backend доверяет `X-Forwarded-For`, `X-Forwarded-Proto` и аналогичным headers только от явно настроенных trusted proxies.

Прямой клиентский header не должен подменять source IP, scheme или host.

### 11.3 CORS

CORS использует allowlist конкретных production origins. `Access-Control-Allow-Origin: *` запрещён для credentialed requests.

Разрешённые methods и headers должны быть минимальны. Origin отражается только после точного совпадения с allowlist.

### 11.4 CSRF

Если аутентификационные cookies автоматически прикладываются браузером к state-changing requests, обязательна CSRF-защита.

Минимальная защита включает:

- `SameSite` cookie policy;
- проверку `Origin` для mutation requests;
- CSRF token для схем, где `SameSite` недостаточно;
- запрет изменения состояния через GET, HEAD и OPTIONS.

Refresh и logout endpoint должны учитывать CSRF threat model.

### 11.5 Security headers

Frontend и API должны устанавливать применимые headers:

- `Content-Security-Policy`;
- `Strict-Transport-Security` в production после проверки HTTPS deployment;
- `X-Content-Type-Options: nosniff`;
- `Referrer-Policy`;
- `Permissions-Policy`;
- `frame-ancestors` через CSP;
- `Cache-Control: no-store` для authentication, session и чувствительных ответов.

Использование устаревшего `X-XSS-Protection` не заменяет CSP и output encoding.

### 11.6 Валидация входных данных

Все DTO валидируются на backend по allowlist-подходу:

- тип;
- обязательность;
- длина;
- формат;
- диапазон;
- допустимое enum-значение;
- количество элементов;
- согласованность полей;
- максимальный размер body.

Неизвестные поля в mutation DTO должны отклоняться, если обратная совместимость не требует иного.

Запрещён mass assignment domain/entity object из JSON.

### 11.7 JSON parsing

JSON decoder должен:

- ограничивать размер body до декодирования;
- отклонять неизвестные поля для команд;
- отклонять несколько JSON-значений в одном body;
- корректно различать пустой body и синтаксическую ошибку;
- не раскрывать внутренние типы или stack trace.

### 11.8 Ошибки

Внешняя ошибка не должна раскрывать:

- SQL;
- имя таблицы или constraint, если это не безопасный контракт;
- stack trace;
- путь файловой системы;
- секреты;
- внутренние идентификаторы сессий;
- детали проверки пароля или токена.

Сравнение доменных/application ошибок выполняется через `errors.Is()` или `errors.As()`.

Маппинг ошибки в HTTP-ответ должен быть централизован в общем response/error handler. Ручное сравнение строк и копирование error mapping по handler-ам запрещено.

### 11.9 Rate limiting

Rate limits должны различаться по риску:

- login, refresh, password reset — строгие;
- public search — отдельные limits и защита от scraping;
- imports/exports — низкая частота и ограничение concurrency;
- mutation commands — защита от burst и accidental retry;
- administrative endpoints — низкие limits и усиленный аудит.

Rate limiting должен учитывать корректное определение client IP через trusted proxy configuration.

### 11.10 Request correlation

Каждый запрос получает server-generated request ID. Входной request ID может быть принят только после валидации формата и не должен позволять log injection.

Request ID возвращается клиенту и связывает application logs, audit events и trace.

## 12. Идемпотентность и replay protection

### 12.1 Область применения

Idempotency key обязателен для критических команд, создающих складской, финансовый или необратимый административный эффект.

Минимально:

- проведение поступления;
- проведение продажи;
- проведение возврата;
- списание;
- корректировка;
- операции импорта, создающие данные;
- опасные административные команды, если повтор может создать дополнительный эффект.

### 12.2 Scope

Idempotency record должен быть scoped как минимум по:

- actor или client identity;
- pharmacy ID для pharmacy-scoped operations;
- operation name;
- idempotency key.

Один и тот же key в другой аптеке или для другой операции не должен случайно коллидировать.

### 12.3 Payload fingerprint

Backend формирует canonical semantic payload и криптографический hash.

Повтор с тем же key и тем же payload возвращает исходный результат.

Повтор с тем же key, но другим semantic payload, отклоняется как conflict и создаёт security/audit signal.

Канонизация не должна зависеть от порядка JSON-полей, незначащих пробелов или client-generated timestamps, не являющихся частью бизнес-смысла.

### 12.4 Транзакционность

Claim idempotency key, выполнение бизнес-операции, запись результата и audit event должны быть согласованы одной транзакцией или формально описанным протоколом.

Нельзя сохранять успешный idempotency result до commit бизнес-изменения.

### 12.5 Replay access control

Даже при существующем успешном idempotency record backend обязан повторно проверить актуальное право actor получить сохранённый результат.

Idempotent replay не должен обходить блокировку пользователя или отзыв назначения.

## 13. Конкурентность и целостность

Security зависит от корректной конкурентной модели.

Критические use case должны:

- блокировать изменяемые агрегаты в фиксированном порядке;
- повторно проверять остаток и состояние после получения lock;
- не выполнять check-then-act вне транзакции;
- использовать database constraints как последнюю линию защиты;
- корректно обрабатывать serialization/deadlock retry без повторного эффекта;
- не доверять cached authorization для mutation commit.

Особенно тестируются гонки:

- две продажи одного остатка;
- продажа одновременно с блокировкой аптеки или пользователя;
- изменение назначения одновременно с операцией аптекаря;
- два refresh request одного token generation;
- повтор одной idempotent command;
- возврат и сторнирование одного документа;
- публикация и повторная модерация каталога.

## 14. Защита данных

### 14.1 Data minimization

Система собирает и хранит только данные, необходимые для бизнес-функции, безопасности, аудита и соблюдения требований.

Нельзя добавлять персональные поля «на будущее» без владельца, цели, срока хранения и правил доступа.

### 14.2 Классификация

Данные классифицируются как:

- public;
- internal;
- confidential;
- secret.

Secret включает raw tokens, signing keys, encryption keys, database credentials и external integration credentials.

### 14.3 Encryption in transit

Все внешние соединения используют TLS. Для production следует использовать TLS и для соединения backend → PostgreSQL, особенно при размещении вне одного защищённого узла.

### 14.4 Encryption at rest

Шифрование дисков, backup storage и managed database storage обязательно для production.

Field-level encryption применяется только для данных, которым недостаточно disk encryption и access control. Решение должно учитывать rotation, searchable fields, backup restore и потерю ключа.

Пароли и refresh tokens не шифруются, а хэшируются.

### 14.5 Резервные копии

Backup содержит те же чувствительные данные, что и production БД, и защищается не слабее.

Обязательны:

- encryption;
- отдельный доступ;
- журналирование операций;
- retention policy;
- регулярная проверка restore;
- запрет использования production backup в development без обезличивания и отдельного разрешения.

### 14.6 Retention и удаление

Срок хранения audit, security logs, sessions, imports и персональных данных должен быть определён до production.

Удаление пользователя не должно разрушать юридически или операционно значимую историю. Вместо физического удаления используется архивирование и, где допустимо, обезличивание отображаемых персональных полей.

## 15. PostgreSQL security

### 15.1 Учётные записи БД

Должны использоваться отдельные credentials как минимум для:

- migration role;
- runtime backend role;
- backup role;
- read-only diagnostics/analytics role при необходимости.

Runtime role не должен иметь права создавать/удалять schema, управлять ролями или выполнять unrestricted DDL.

Migration role не используется приложением в runtime.

### 15.2 Network access

PostgreSQL не публикуется в публичный интернет. Доступ разрешается только из backend runtime, migration job и контролируемых administrative channels.

### 15.3 SQL injection

Все runtime SQL queries используют параметры. Конкатенация пользовательских значений в SQL запрещена.

Динамические identifiers допускаются только через allowlist и безопасное quoting API; параметры SQL не заменяют identifiers.

### 15.4 Constraints

Критические инварианты дублируются constraints, где это возможно:

- non-negative quantity;
- допустимые enum/status values;
- uniqueness;
- foreign keys;
- одна активная связь, если это правило модели;
- корректные временные диапазоны;
- append-only protection для immutable records через revoke privileges и при необходимости trigger.

Constraints не заменяют domain validation, но защищают от ошибок и обхода application layer.

### 15.5 Прямой доступ

Прямое изменение production business data через SQL запрещено как штатная операция.

Emergency change требует:

- incident/change ticket;
- peer review;
- backup или точку восстановления;
- минимальный scoped script;
- запись исполнителя и причины;
- post-change verification;
- отражение в audit/change log.

## 16. Неизменяемость и аудит

### 16.1 Audit scope

Audit event обязателен для:

- успешных и отклонённых login/refresh/logout security событий;
- создания, блокировки, архивирования пользователя;
- изменения роли;
- назначения и отзыва назначения аптекаря;
- создания, блокировки и архивирования аптеки;
- изменения цен;
- проведения и сторнирования поступлений, продаж, возвратов, списаний и корректировок;
- импорта и модерации каталога;
- публикации глобальной карточки;
- просмотра или экспорта особенно чувствительных данных;
- изменения security configuration;
- административного отзыва сессий;
- обнаружения refresh token reuse;
- idempotency payload conflict;
- emergency/manual data operation.

### 16.2 Содержимое audit event

Минимально:

- event ID;
- server timestamp;
- actor type и actor ID;
- session ID, если применимо;
- action;
- target type и target ID;
- pharmacy ID, если применимо;
- result: success/denied/failure;
- reason code;
- request ID;
- source IP или безопасно нормализованный network identifier;
- user-agent в ограниченном размере;
- correlation/trace ID;
- structured metadata без секретов;
- before/after snapshot только для явно разрешённых полей.

Audit metadata не должна содержать пароль, raw token, полный authorization header, secret, database DSN или неограниченный request body.

### 16.3 Транзакционность аудита

Для бизнес-операции, которая считается успешной только при наличии аудита, audit event создаётся в той же транзакции до commit.

Если audit insert не выполнен, операция должна завершиться ошибкой и rollback.

Технические события, возникающие до открытия business transaction, могут записываться в отдельный security log, но это не заменяет transactional audit.

### 16.4 Append-only

Audit events, inventory movements и проведённые документы являются append-only.

Application runtime не получает DELETE/UPDATE privileges на audit events и immutable movement records, кроме строго определённых технических полей, если они вообще необходимы.

Исправление выполняется новым компенсирующим событием, а не переписыванием истории.

### 16.5 Доступ к аудиту

Audit доступен только уполномоченным ADMIN и ограниченным техническим ролям.

Просмотр audit также аудируется. Массовый экспорт требует повышенного контроля, reason и повторной аутентификации.

## 17. Логи, метрики и трассировка

### 17.1 Structured logging

Backend использует structured logs с фиксированными именами полей.

Логи должны позволять связать request, actor, session, pharmacy, use case и результат, но не раскрывать secret values.

### 17.2 Запрещённые данные в логах

Запрещено логировать:

- пароли;
- raw access/refresh/reset tokens;
- Authorization и Cookie headers;
- database credentials;
- encryption/signing keys;
- полный импортируемый файл;
- полный request/response body без явного безопасного allowlist;
- чувствительные персональные данные без необходимости.

### 17.3 Log injection

Пользовательские строки должны записываться как structured values, а не конкатенироваться в свободный текст. Control characters и чрезмерная длина нормализуются или ограничиваются.

### 17.4 Security monitoring

Минимальные сигналы:

- всплеск failed login;
- refresh token reuse;
- большое число 401/403;
- частые idempotency conflicts;
- необычная активность ADMIN;
- массовый экспорт;
- попытки доступа PHARMACIST к чужой аптеке;
- повторяющиеся validation failures;
- изменение роли или назначения;
- отключение/ошибка audit pipeline;
- неожиданное изменение объёма складских корректировок;
- ошибки проверки подписи токена;
- рост rate-limit rejects.

## 18. Frontend security

### 18.1 Общие правила

Frontend является недоверенным клиентом, но обязан снижать риск компрометации пользователя.

### 18.2 XSS

Требования:

- не использовать небезопасный HTML rendering без sanitization;
- не вставлять пользовательские строки через raw HTML API;
- применять контекстное escaping;
- использовать строгую CSP;
- минимизировать third-party scripts;
- не передавать токены analytics и error tracking systems;
- проверять зависимости и lockfile.

### 18.3 Token handling

Frontend не должен:

- сохранять refresh token в JS-accessible storage;
- выводить token в console;
- включать token в URL;
- помещать token в error report;
- сохранять чувствительные ответы дольше необходимого.

При logout, session expiry или смене пользователя in-memory security state очищается.

### 18.4 Route guards

Route guard улучшает UX, но не является security boundary. Backend выполняет полную авторизацию каждого запроса.

### 18.5 UI для опасных действий

Для опасных административных и корректирующих действий frontend должен:

- явно показывать цель и последствия;
- требовать подтверждение;
- не использовать неразличимые destructive controls;
- передавать reason, если он обязателен;
- корректно обрабатывать stale state и conflict;
- не повторять mutation автоматически без idempotency key.

## 19. Импорт файлов

### 19.1 Общая модель

Импортируемый файл считается недоверенным независимо от роли загрузившего пользователя.

### 19.2 Ограничения

Обязательны:

- allowlist MIME/type и расширения;
- проверка реального формата, а не только имени;
- ограничение размера;
- ограничение количества строк, столбцов и длины ячейки;
- безопасный streaming parser;
- timeout и memory limits;
- случайное server-side имя;
- хранение вне executable/static directories;
- запрет path traversal;
- отсутствие интерпретации формул как кода;
- защита от CSV/Spreadsheet formula injection при последующем экспорте;
- staging и валидация до публикации;
- аудит загрузки, модерации и публикации.

### 19.3 Архивы

Если архивные форматы будут поддержаны, обязательны ограничения на количество файлов, суммарный распакованный размер, глубину вложенности и compression ratio для защиты от zip bomb.

## 20. Экспорт данных

Экспорт чувствительных данных должен:

- иметь отдельное permission;
- ограничивать scope;
- требовать reason для административного массового экспорта;
- использовать server-side фильтрацию;
- иметь лимит объёма;
- защищаться от CSV formula injection;
- не включать секреты и скрытые поля;
- создавать audit event;
- выдавать файл через одноразовую или короткоживущую ссылку, если используется asynchronous export.

## 21. Внешние интеграции и SSRF

Любая будущая исходящая интеграция должна использовать отдельный adapter и allowlist destinations.

Запрещено принимать произвольный URL от клиента и выполнять server-side request без строгой политики.

HTTP client должен иметь:

- connect/read/overall timeout;
- ограничение response size;
- запрет неожиданных redirect или их строгую проверку;
- DNS/IP validation при SSRF-риске;
- запрет доступа к loopback, link-local, metadata endpoints и private networks, если это не разрешено явно;
- TLS verification;
- ограниченный набор headers;
- circuit breaker или bounded retry для некритических integrations.

## 22. Секреты и криптографические ключи

### 22.1 Хранение

Production secrets не хранятся:

- в Git;
- в frontend bundle;
- в Docker image layer;
- в CI logs;
- в `.env` файле, распространяемом как артефакт;
- в документации или task tracker.

Секреты предоставляются через secret manager или защищённый runtime mechanism.

### 22.2 Разделение

Отдельные environment используют разные secrets. Development credentials не действуют в production.

Отдельные назначения должны по возможности использовать разные ключи:

- token signing;
- field encryption;
- CSRF/HMAC;
- external integrations;
- database access.

Один универсальный secret запрещён.

### 22.3 Ротация

Для каждого секрета определяются owner, назначение, срок, процедура ротации и действия при компрометации.

Token signing key rotation должна поддерживать одновременную проверку ограниченного числа активных key IDs. Новый ключ используется для подписи, старый — только для проверки до истечения допустимых токенов.

### 22.4 Криптографические требования

Используются стандартные, поддерживаемые библиотеки и алгоритмы. Самодельная криптография запрещена.

Random IDs, tokens и nonces генерируются CSPRNG.

Сравнение secret-derived values выполняется constant-time функцией, когда это применимо.

## 23. Конфигурация и безопасный startup

Приложение должно завершать startup с ошибкой при:

- отсутствующем обязательном secret;
- placeholder/default production secret;
- небезопасной длине signing key;
- некорректном token TTL;
- wildcard CORS для credentialed production API;
- отключённом TLS expectation в production deployment;
- включённом debug mode;
- неизвестной environment;
- невозможности подключиться к обязательной БД;
- несовместимой версии schema;
- конфликтующих security settings.

Конфигурация валидируется централизованно и не логируется целиком.

## 24. Dependency и supply-chain security

Обязательны:

- version pinning и lockfiles;
- dependency review для новых библиотек;
- автоматическое сканирование известных уязвимостей;
- обновление критических vulnerabilities в установленный SLA;
- минимизация зависимостей;
- запрет неподдерживаемых библиотек;
- проверка лицензий;
- signed или provenance-aware build artifacts по мере зрелости pipeline;
- container image scanning;
- секрет-сканирование репозитория;
- branch protection и обязательный review для security-critical changes.

Dependency scanner не заменяет code review и threat modeling.

## 25. Container и runtime hardening

Production container должен:

- запускаться не от root;
- использовать минимальный runtime image;
- не содержать compiler, shell и package manager без необходимости;
- иметь read-only root filesystem, где возможно;
- использовать отдельный writable temp volume;
- не включать source code и build secrets;
- ограничивать Linux capabilities;
- иметь memory/CPU/pid limits;
- корректно обрабатывать SIGTERM;
- публиковать только необходимые ports.

Backend management/debug endpoints не должны быть доступны из публичного ingress.

## 26. Безопасная разработка

### 26.1 Security review triggers

Отдельный security review обязателен при изменении:

- authentication/session/token flow;
- ролей, permissions или assignments;
- password/reset/MFA flow;
- критических transaction boundaries;
- idempotency;
- audit;
- импорта/экспорта;
- криптографии и secrets;
- CORS/CSRF/CSP;
- public/private data boundary;
- внешней интеграции;
- deployment topology;
- backup/restore;
- административных операций.

### 26.2 Pull request requirements

Security-critical PR должен содержать:

- описание угрозы или abuse case;
- изменённые security assumptions;
- тесты positive/negative path;
- проверку horizontal и vertical authorization;
- проверку audit;
- проверку отсутствия secret leakage;
- обновление `05-api-design.md`, если изменён HTTP contract;
- обновление этого документа или ADR, если изменена модель безопасности.

### 26.3 Запрещённые практики

Запрещены:

- сравнение ошибок по строке;
- ручной разрозненный HTTP error mapping;
- хранение plaintext passwords/tokens;
- доверие role claim без необходимой server-side revalidation;
- универсальный admin update endpoint;
- generic repository `Save(entity)` для критических агрегатов, позволяющий менять произвольное состояние;
- автоматический retry неидемпотентной команды;
- logging полного request body по умолчанию;
- использование `gin.Context`, `pgx.Tx` или HTTP types в domain/application contracts;
- отключение TLS verification;
- `Access-Control-Allow-Origin: *` с credentials;
- secrets в Git и frontend environment variables;
- silent fallback на небезопасную конфигурацию.

## 27. Security testing

### 27.1 Обязательные категории

Должны быть автоматизированы:

- authentication success/failure;
- blocked/archived user denial;
- session expiry и revoke;
- refresh rotation;
- refresh reuse detection;
- logout idempotency;
- role matrix;
- horizontal access между аптеками;
- vertical privilege escalation;
- stale authorization внутри транзакции;
- unknown role/status fail-closed;
- CSRF protection;
- CORS allowlist;
- security headers;
- request size limits;
- unknown JSON fields;
- SQL injection payloads;
- XSS-relevant rendering tests;
- idempotency replay и payload conflict;
- race/concurrency tests;
- audit success и audit failure rollback;
- log redaction;
- import malicious/oversized files;
- CSV formula injection;
- secret/config validation;
- database privilege expectations;
- migration tests для security constraints.

### 27.2 Authorization matrix

Для каждого endpoint в `05-api-design.md` должна быть определена матрица:

- public/CLIENT/PHARMACIST/ADMIN;
- allowed/denied;
- resource scope;
- assignment requirement;
- reauthentication requirement;
- audit requirement;
- idempotency requirement;
- sensitive response classification.

Каждый denied case должен иметь тест, а не только allowed path.

### 27.3 Adversarial tests

Необходимо тестировать:

- подмену route ID;
- подмену `pharmacy_id` в body;
- лишние JSON-поля `role`, `status`, `total`, `created_at`;
- replay украденного refresh token;
- два параллельных refresh;
- повтор mutation после timeout клиента;
- blocked user с ещё не истёкшим access token;
- revoked assignment во время операции;
- попытку изменить immutable document;
- malformed JWT и неизвестный `kid`;
- oversized headers/body;
- malicious filename/path;
- duplicate JSON keys, если parser допускает неоднозначность;
- Unicode confusables в идентификаторе входа согласно выбранной политике.

### 27.4 Manual testing

Перед production release выполняются:

- threat model review;
- dependency and container scan;
- проверка deployment configuration;
- backup restore exercise;
- access review;
- secret rotation exercise;
- targeted penetration test для authentication, authorization и critical business flows.

## 28. Реагирование на инциденты

### 28.1 Минимальный процесс

Для security incident должны быть определены:

1. обнаружение и классификация;
2. containment;
3. сохранение evidence;
4. отзыв сессий и ротация secrets;
5. исправление причины;
6. проверка целостности данных;
7. восстановление;
8. уведомление заинтересованных лиц согласно применимым требованиям;
9. post-incident review;
10. обновление controls, tests и документации.

### 28.2 Компрометация аккаунта

Поддерживаются действия:

- блокировка пользователя;
- отзыв всех sessions;
- обязательная смена пароля;
- сброс MFA;
- просмотр security events;
- проверка действий пользователя за период;
- маркировка инцидента и связь с audit records.

### 28.3 Компрометация signing key

Необходимо:

- прекратить подпись скомпрометированным ключом;
- выпустить новый key ID;
- отозвать затронутые sessions или tokens согласно масштабу;
- проверить source и pipeline compromise;
- зафиксировать incident timeline;
- выполнить контролируемую ротацию зависимых секретов.

### 28.4 Нарушение целостности данных

При подозрении на прямое изменение БД:

- ограничить запись;
- сохранить WAL/log/evidence;
- сравнить audit, immutable movements и backup;
- определить affected aggregates;
- не «исправлять» историю до завершения фиксации evidence;
- применять только утверждённые compensating operations или controlled restore.

## 29. Security SLA и регулярные проверки

До production должны быть утверждены конкретные SLA минимум для:

- отзыва доступа blocked user;
- исправления critical/high vulnerabilities;
- ротации скомпрометированного secret;
- хранения audit/security logs;
- реагирования на alert;
- восстановления из backup;
- review активных ADMIN и assignments;
- удаления истёкших sessions и temporary imports.

Рекомендуемый минимум:

- review активных ADMIN — ежемесячно;
- review PHARMACIST assignments — ежеквартально и при кадровом изменении;
- restore test — не реже ежеквартально;
- secret rotation exercise — не реже ежегодно;
- access review production infrastructure — ежеквартально;
- targeted security test — перед каждым major release.

## 30. Security requirements для модулей

### 30.1 `identity`

- password hashing;
- session lifecycle;
- refresh rotation и reuse detection;
- account state;
- role management;
- MFA hooks;
- security events;
- отсутствие raw token storage.

### 30.2 `pharmacy`

- assignment как authoritative access scope;
- запрет доверия client-supplied pharmacy ID;
- блокировка/архивирование аптеки;
- аудит изменения public profile и assignments.

### 30.3 `catalog`

- staging для imports;
- allowlisted publication flow;
- защита от mass assignment;
- аудит merge/moderation/publication;
- sanitization отображаемых текстов.

### 30.4 `assortment`

- pharmacy scope;
- backend price validation;
- история изменения цены;
- запрет скрытого изменения historical snapshots.

### 30.5 `inventory`

- append-only movements;
- non-negative stock;
- row locking;
- запрет продажи просроченных/карантинных лотов;
- controlled adjustments с reason;
- audit в одной транзакции.

### 30.6 `sales`

- server-side totals;
- FEFO allocation;
- idempotency;
- concurrency protection;
- immutable posted sale;
- compensating operation вместо edit/delete.

### 30.7 `returns`

- связь с исходной продажей;
- защита от превышения возвращаемого количества;
- отдельная проверка допустимости возврата в saleable stock;
- юридически утверждённая policy;
- audit и idempotency.

### 30.8 `search`

- только публичная проекция;
- отсутствие служебных остатков и закупочных цен;
- rate limiting;
- минимизация location data;
- защита от дорогостоящих запросов.

### 30.9 `audit`

- append-only;
- отдельные write/query ports;
- отсутствие generic update/delete;
- ограниченный доступ;
- transactional integration с critical use cases.

## 31. Обязательные ADR

До завершения production-ready реализации должны быть приняты или подтверждены ADR для:

1. формата access token и алгоритма подписи;
2. refresh token rotation и session family reuse detection;
3. browser token storage и CSRF model;
4. password hashing algorithm и параметры;
5. MFA для ADMIN;
6. transactional audit и append-only enforcement;
7. secret management и signing key rotation;
8. production database roles;
9. retention персональных, audit и security данных;
10. emergency production data change process.

## 32. Критерии готовности функции

Security-sensitive feature считается готовой, если:

1. определены assets, actor и abuse cases;
2. endpoint внесён в `05-api-design.md`;
3. определена authorization matrix;
4. backend не доверяет client-calculated values и scope;
5. mutation имеет idempotency, если повтор создаёт дополнительный эффект;
6. критическая проверка выполняется в транзакции;
7. audit event записывается согласно policy;
8. ошибки проходят через централизованный mapper и сравниваются через `errors.Is()`/`errors.As()`;
9. sensitive data не попадает в логи и ответы;
10. есть positive, negative, horizontal, vertical и concurrency tests;
11. конфигурация имеет безопасные defaults и startup validation;
12. документация и ADR синхронизированы.

## 33. Открытые вопросы

До production необходимо закрыть:

1. юридические правила возврата лекарств и обработки рецептурных данных;
2. точные сроки хранения audit, security logs и персональных данных;
3. обязательный MFA mechanism для ADMIN;
4. конкретные token TTL и session limits;
5. password hash algorithm и параметры для production hardware;
6. необходимость field-level encryption для отдельных данных;
7. допустимость точного хранения IP-адресов и срок retention;
8. политика повторного использования login identifier;
9. перечень действий, требующих four-eyes approval;
10. место хранения secrets и KMS/secret manager;
11. production topology TLS между ingress, backend и PostgreSQL;
12. security SLA и ответственные лица.

## 34. Итоговые обязательные правила

1. Клиент, токен и route parameter не являются достаточным источником полномочий.
2. Backend повторно проверяет active user, role и pharmacy assignment для критических операций.
3. Критическая авторизация и бизнес-изменение выполняются в одной транзакционной границе.
4. Access token короткоживущий; refresh token хранится только как hash server-side и ротируется при каждом использовании.
5. Повторное использование refresh token отзывает session family.
6. Raw passwords, tokens и secrets никогда не хранятся и не логируются.
7. ADMIN — высокорисковая роль с усиленным аудитом, повторной аутентификацией и MFA target.
8. Проведённые документы, движения и audit append-only; исправления выполняются компенсирующими операциями.
9. Каждая критическая команда имеет корректную idempotency и replay authorization.
10. Error mapping централизован; ошибки сравниваются только через `errors.Is()`/`errors.As()`.
11. Public API использует только специально подготовленную публичную проекцию.
12. Production secrets не находятся в Git, frontend bundle, image layer или logs.
13. Security controls должны быть выражены в коде, транзакциях, constraints и автоматических тестах.
14. Любая неоднозначность в authorization или security state завершается отказом в доступе.
