# ADR-0015: Логирование, конфигурация, HTTP middleware и локальная персистентность

- **Status:** Accepted
- **Date:** 2026-07-17
- **Decision owners:** PharmacyCRM team

## Context

PharmacyCRM должна быть диагностируемой как при локальной разработке, так и при запуске в Docker. Для расследования ошибок недостаточно только вывода в терминал: структурированные технические логи должны одновременно сохраняться на локальной машине. Данные PostgreSQL также не должны теряться при пересоздании контейнера.

HTTP-запрос проходит через несколько поперечных технических механизмов: восстановление после panic, назначение request ID, распределённую трассировку, access logging, аутентификацию и авторизацию. Их состав и порядок должны быть централизованы, иначе разные группы маршрутов будут иметь различное поведение и неполные диагностические данные.

Конфигурация приложения должна загружаться из переменных окружения в типизированную структуру. Для этого выбран `github.com/kelseyhightower/envconfig`. Согласно официальной документации библиотеки, конфигурация обрабатывается через `envconfig.Process(prefix, &config)`, а поля могут использовать теги `required`, `default`, `envconfig` и `split_words`. Библиотека читает переменные окружения процесса и сама по себе не является загрузчиком файла `.env`.

## Decision

### 1. Библиотека логирования

В backend используется `go.uber.org/zap`, а прикладной runtime получает настроенный `*zap.Logger` из Composition Root.

Выбор конкретного логгера является инфраструктурным решением:

- Domain не импортирует `zap` и не выполняет техническое логирование;
- Application не должен зависеть от `*zap.Logger` как от обязательной части бизнес-контракта;
- Delivery, Infrastructure, workers и bootstrap могут использовать `zap` напрямую либо через узкий внутренний интерфейс, если это повышает тестируемость;
- бизнес-аудит не заменяется техническими логами Zap.

Для production-подобных окружений используется JSON encoding. Для локальной разработки допускается console encoding в терминале, но файл должен сохранять структурированный формат, пригодный для автоматического анализа.

### 2. Двойной вывод логов

Одна запись технического лога должна направляться одновременно:

1. в `stdout`/`stderr` контейнера;
2. в файл внутри каталога логов контейнера, примонтированного к локальной машине.

Концептуально:

```text
zap.Logger
  ├─ console core -> stdout/stderr
  └─ file core    -> /var/log/pharmacycrm/app.log
                         |
                         └─ Docker volume/bind mount -> локальный каталог
```

Рекомендуемый путь внутри контейнера:

```text
/var/log/pharmacycrm/
```

Конкретный локальный путь и тип монтирования фиксируются в deployment-документе. Каталог логов, файлы логов и иные runtime-артефакты не коммитятся в Git.

Файловый sink обязан поддерживать ротацию, чтобы бесконтрольный рост файла не исчерпал диск. Должны настраиваться:

- максимальный размер одного файла;
- максимальное число архивов;
- максимальный возраст;
- сжатие архивов;
- политика поведения при невозможности записи файла.

Невозможность записи файлового лога не должна незаметно отключать вывод в терминал. Ошибка инициализации обязательного sink должна либо завершать запуск fail-fast, либо явно переводить приложение в документированный деградированный режим.

При graceful shutdown вызывается `logger.Sync()`. Ошибки `Sync` для неподдерживающих синхронизацию терминальных дескрипторов обрабатываются осознанно, а не скрываются общим игнорированием всех ошибок.

### 3. Формат технического лога

Время записывается в локально согласованной временной политике проекта, предпочтительно UTC, в ISO 8601/RFC 3339 с точностью не хуже миллисекунд.

Минимальные общие поля:

| Поле | Назначение |
|---|---|
| `timestamp` | дата и время с миллисекундами |
| `level` | уровень события |
| `message` | краткое стабильное сообщение |
| `service` | имя приложения или worker |
| `environment` | окружение запуска |
| `request_id` | идентификатор HTTP-запроса, если применимо |
| `trace_id` | идентификатор trace, если tracing активен |
| `span_id` | идентификатор span, если применимо |
| `user_id` | идентификатор актёра после успешной аутентификации |
| `pharmacy_id` | scope аптеки, если он установлен и проверен |
| `error` | безопасное описание ошибки |
| `duration_ms` | длительность операции в миллисекундах |

Имена сообщений и полей должны быть достаточно стабильными для поиска и построения метрик. Форматированная строка не заменяет структурированные поля.

### 4. HTTP access logging

Access log формируется централизованным Gin middleware и содержит как минимум:

- timestamp с точностью до миллисекунд;
- HTTP method — тип запроса;
- route pattern, например `/api/v1/pharmacies/:id`, а не только фактический URL;
- безопасный request URI;
- client IP с учётом явно настроенных trusted proxies;
- protocol;
- request size;
- response status code;
- response size;
- duration в миллисекундах;
- request ID;
- trace ID и span ID при наличии;
- идентификатор пользователя после аутентификации;
- итоговое сообщение;
- категорию ошибки, если запрос завершился ошибкой.

Для диагностических endpoint-ов может регистрироваться безопасное представление request и response body, однако полное безусловное логирование тел запрещено.

Причины:

- запросы могут содержать пароли, токены, персональные данные и рецептурную информацию;
- ответы могут содержать чувствительные административные или пользовательские данные;
- большие тела могут вызвать избыточное потребление памяти и диска;
- binary/multipart payload не предназначен для журналирования;
- повторное чтение body может нарушить обработку запроса, если middleware реализован неправильно.

При включённом body logging обязательны:

1. allowlist маршрутов и content type;
2. ограничение максимального журналируемого размера;
3. маскирование как минимум `password`, `token`, `access_token`, `refresh_token`, `authorization`, `cookie`, `set-cookie`, секретов и иных чувствительных полей;
4. полное исключение `Authorization`, cookies и файлового содержимого;
5. признак `body_truncated`, если содержимое сокращено;
6. настройка уровня по окружению;
7. тесты, подтверждающие отсутствие секретов в логах.

Для production базовым режимом является логирование metadata и бизнес-безопасного summary, а не полного body.

### 5. Panic recovery

Recovery middleware обязан:

- перехватывать panic на HTTP-границе;
- сохранять request ID и trace context;
- логировать panic на уровне `Error` или `DPanic` согласно конфигурации;
- включать stack trace в технический лог;
- не возвращать stack trace, SQL, путь к файлам или внутреннее сообщение клиенту;
- возвращать единый API error envelope со статусом `500 Internal Server Error`, если ответ ещё не был отправлен;
- корректно учитывать частично записанный HTTP-ответ;
- не скрывать ошибки и panic фоновых goroutine, которые не находятся в контексте запроса.

Recovery не заменяет rollback: Unit of Work обязан самостоятельно гарантировать rollback при panic внутри транзакционного callback.

### 6. Request ID

Request ID middleware выполняется до access logging, tracing, аутентификации и recovery-логирования, чтобы идентификатор присутствовал во всех последующих событиях.

Правила:

- входной `X-Request-ID` принимается только после проверки формата и длины;
- при отсутствии или недопустимости значения backend генерирует новый идентификатор;
- идентификатор помещается в `context.Context` и Gin context только на delivery-границе;
- `X-Request-ID` возвращается в response header;
- идентификатор включается во все логи запроса и передаётся во внешние вызовы, где это безопасно;
- request ID не используется как секрет, токен доступа или idempotency key.

### 7. Tracing middleware

Tracing является отдельным механизмом от request ID.

Middleware должен:

- извлекать валидный trace context из поддерживаемых заголовков либо начинать новый server span;
- записывать HTTP method, route pattern, status code и duration;
- не помещать секреты и полные тела в span attributes;
- передавать `context.Context` use case и инфраструктурным адаптерам;
- завершать span после формирования ответа;
- помечать неожиданные ошибки и panic;
- поддерживать отключение exporter без изменения бизнес-кода.

Конкретная реализация tracing и exporter фиксируется в observability-документе. OpenTelemetry является предпочтительной нейтральной абстракцией, но его подключение не должно протекать в Domain.

### 8. Порядок HTTP middleware

Централизованный bootstrap в `internal/platform/httpserver` должен явно устанавливать порядок middleware.

Базовый порядок внешних технических middleware:

1. request ID;
2. panic recovery;
3. tracing;
4. structured access logging;
5. trusted proxy handling;
6. security headers;
7. CORS;
8. request size limits;
9. rate limiting для соответствующих маршрутов;
10. authentication parsing;
11. route-level authorization/access middleware;
12. handler.

Точная композиция может быть разделена на global, group и route middleware, но request ID, recovery, tracing и access logging должны охватывать защищённые и публичные маршруты.

Access logging должно фиксировать окончательный status code и duration, поэтому middleware выполняет запись после `c.Next()`. Recovery должен быть расположен так, чтобы panic также попадал в итоговый access log и отдельный error log.

### 9. Конфигурация через `.env` и `envconfig`

Локальная конфигурация хранится в файле `.env`, который не коммитится в Git. В репозитории хранится безопасный `.env.example` без реальных секретов.

Важно разделять два механизма:

- `.env` или Docker Compose `env_file` помещает значения в окружение процесса;
- `github.com/kelseyhightower/envconfig` читает окружение процесса и заполняет типизированную Go-структуру.

Приложение не должно предполагать, что `envconfig` самостоятельно разбирает файл `.env`.

Используется единая конфигурационная структура либо набор вложенных структур, загружаемых в Composition Root:

```go
type Config struct {
    Environment string `envconfig:"APP_ENV" required:"true"`
    HTTP        HTTPConfig
    Database    DatabaseConfig
    Logging     LoggingConfig
    Tracing     TracingConfig
}

func Load() (Config, error) {
    var cfg Config
    if err := envconfig.Process("PHARMACYCRM", &cfg); err != nil {
        return Config{}, fmt.Errorf("load environment configuration: %w", err)
    }
    if err := cfg.Validate(); err != nil {
        return Config{}, fmt.Errorf("validate configuration: %w", err)
    }
    return cfg, nil
}
```

Имена и struct tags проверяются по официальной документации `envconfig` перед реализацией. Следует использовать:

- `required:"true"` для обязательных значений;
- `default:"..."` только для безопасных неproduction defaults;
- `split_words:"true"` или явный `envconfig:"NAME"`, когда это делает имя переменной однозначным;
- `time.Duration` для timeout и interval;
- пользовательский decoder только при реальной необходимости.

После `envconfig.Process` выполняется отдельная семантическая валидация: диапазоны портов, допустимые уровни логирования, обязательность секретов, корректность URL, положительность timeout и согласованность параметров.

Пустое значение обязательной переменной не должно считаться допустимым только потому, что библиотека обнаружила её присутствие. Конфигурационный валидатор обязан отдельно отклонять пустые обязательные строки. Это важно, поскольку официальная документация `envconfig` указывает, что `required:"true"` проверяет отсутствие переменной, но присутствующее пустое значение само по себе не вызывает ошибку.

Runtime-модули не вызывают `os.Getenv` и `envconfig.Process` самостоятельно. Готовая неизменяемая конфигурация передаётся из Composition Root.

### 10. PostgreSQL в Docker

PostgreSQL использует постоянный Docker volume или явно настроенный bind mount для каталога данных:

```text
/var/lib/postgresql/data
```

Пересоздание контейнера приложения или PostgreSQL не должно удалять данные, пока оператор явно не удалил volume или локальный каталог.

Для локальной разработки порт PostgreSQL пробрасывается:

```text
host 5433 -> container 5432
```

Приложения внутри Docker Compose обращаются к PostgreSQL по имени сервиса и внутреннему порту `5432`. Порт `5433` предназначен для подключения инструментов с хостовой машины, например IDE, `psql` или клиента администрирования.

Публикация порта БД является настройкой локального/dev-окружения и не должна автоматически переноситься в production. В production PostgreSQL не должен быть доступен из публичной сети без отдельного инфраструктурного решения.

Docker volume защищает от потери данных при пересоздании контейнера, но не является резервной копией. Deployment-документ обязан отдельно определить backup, restore test и политику хранения резервных копий.

### 11. Локальные Docker volumes

Минимально требуются два независимых persistent mount:

| Назначение | Путь внутри контейнера | Пример локального назначения |
|---|---|---|
| PostgreSQL data | `/var/lib/postgresql/data` | Docker named volume или локальный каталог данных |
| Backend logs | `/var/log/pharmacycrm` | локальный каталог логов или named volume |

Разделение обязательно: удаление/очистка логов не должно затрагивать данные БД, а обслуживание БД не должно удалять журналы приложения.

Права файловой системы на каталогах должны соответствовать непривилегированным пользователям контейнеров. Запуск backend от root только ради записи логов запрещён.

## Consequences

### Positive

- структурированные и коррелируемые логи доступны и в терминале, и после остановки контейнера;
- request ID, trace ID и actor context связывают технические события одного запроса;
- panic не завершает весь HTTP-сервер и не раскрывает внутренние детали клиенту;
- типизированная конфигурация загружается централизованно;
- данные PostgreSQL переживают пересоздание контейнера;
- локальные DB-клиенты подключаются через `localhost:5433`;
- архитектура готова к централизованному сбору логов без изменения бизнес-слоёв.

### Negative

- файловое логирование увеличивает потребление диска и требует ротации;
- журналирование request/response требует строгого redaction и тестирования;
- bind mounts различаются между ОС и требуют аккуратного deployment-конфига;
- проброшенный порт БД увеличивает поверхность атаки локальной среды;
- `envconfig` не загружает `.env` самостоятельно, поэтому окружение должен подготовить Docker Compose, shell или отдельный bootstrap-механизм.

## Rejected alternatives

### Только stdout

Отклонено для локального Docker-окружения, поскольку пользователь требует долговременного локального файла. stdout при этом сохраняется как обязательный канал для Docker logs и будущего централизованного сбора.

### Только файловые логи

Отклонено: ухудшает интеграцию с контейнерной платформой и диагностику через `docker logs`.

### Безусловное логирование полных request/response body

Отклонено из-за риска утечки паролей, токенов, персональных данных и неконтролируемого роста объёма логов.

### Хранение конфигурации только в коде

Отклонено: смешивает конфигурацию окружения с артефактом приложения и требует пересборки для изменения runtime-параметров.

### Использование Docker volume как backup

Отклонено: volume обеспечивает персистентность, но не защищает от логической порчи, случайного удаления, повреждения диска или потери хоста.

## Invariants

1. Backend использует `go.uber.org/zap` как технический логгер.
2. Логи одновременно направляются в терминал и persistent файл.
3. Файловые логи имеют ротацию и не коммитятся в Git.
4. Каждая HTTP-запись содержит timestamp с миллисекундами, method, route/request URI, status code, duration и request ID.
5. При активном tracing в записи присутствуют trace ID и span ID.
6. Пароли, токены, cookies, секреты и чувствительные тела не попадают в логи.
7. Полные request/response body не журналируются без allowlist, size limit и redaction.
8. Request ID устанавливается до остальных диагностических middleware и возвращается клиенту.
9. Panic логируется со stack trace, но клиент получает безопасный ответ.
10. `.env` не коммитится; `.env.example` не содержит секретов.
11. `envconfig` вызывается централизованно в Composition Root и читает окружение процесса, а не файл `.env` напрямую.
12. Обязательные пустые строки отклоняются дополнительной валидацией после `envconfig.Process`.
13. PostgreSQL data и backend logs используют разные persistent mounts.
14. Локальный PostgreSQL публикуется как `localhost:5433 -> postgres:5432`.
15. Публикация порта PostgreSQL наружу не считается production-default.
16. Docker volume не считается резервной копией.
17. Domain не зависит от Zap, envconfig, Gin, Docker или файловой системы.
