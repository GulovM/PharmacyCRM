# ADR-0017: Единый каталог HTTP API-контрактов

- **Status:** Accepted
- **Date:** 2026-07-17
- **Decision owners:** PharmacyCRM team

## Context

Frontend PharmacyCRM должен интегрироваться с backend без необходимости изучать handlers, application-use-case, DTO, repository-код и SQL-схему для восстановления фактического HTTP-контракта.

Если endpoint реализован только в коде, но не отражён в едином документе, возникают типовые проблемы:

- frontend использует неверный URL, HTTP method или формат payload;
- разные разработчики по-разному трактуют обязательность и nullable-поля;
- статусы и error codes приходится восстанавливать по реализации;
- изменение backend ломает frontend без заметного изменения документации;
- примеры запросов расходятся с реальным API;
- невозможно быстро определить, какие endpoint-ы уже реализованы.

## Decision

Единым человекочитаемым источником HTTP API-контрактов является:

```text
docs/05-api-design.md
```

Каждая feature, которая добавляет, изменяет или удаляет HTTP endpoint, обязана обновлять этот файл в том же change set.

Feature с HTTP API не считается завершённой, если код, тесты и `docs/05-api-design.md` не согласованы.

Документ предназначен прежде всего для frontend-разработки, QA, интеграционных тестов и сопровождения API.

## Source-of-truth hierarchy

Для внешнего поведения endpoint-а действует следующий порядок:

1. `02-srs.md` — обязательное продуктовое поведение;
2. `05-api-design.md` — конкретный HTTP-контракт;
3. принятые ADR — способ реализации архитектурно значимых аспектов;
4. реализация и тесты.

Код не должен молча расходиться с `05-api-design.md`. Если контракт изменяется, документ обновляется одновременно с кодом.

## Обязательная структура описания endpoint-а

Каждый endpoint должен содержать не менее следующих данных.

### 1. Назначение

Краткое объяснение, какую пользовательскую или системную задачу решает endpoint.

### 2. HTTP contract

- HTTP method;
- полный URL/path;
- версия API;
- content type;
- является ли endpoint публичным или защищённым;
- допустимые роли;
- дополнительные ограничения доступа, например назначение аптекаря конкретной аптеке.

Пример:

```text
POST /api/v1/pharmacies/{pharmacy_id}/sales
```

### 3. Path parameters

Для каждого параметра:

- имя;
- тип;
- формат;
- обязательность;
- смысл;
- ограничения;
- пример.

### 4. Query parameters

Для каждого параметра:

- имя;
- тип;
- обязательность;
- default;
- допустимые значения;
- правила пагинации, сортировки и фильтрации;
- пример.

### 5. Headers

Должны быть указаны применимые заголовки:

- `Authorization`;
- `Content-Type`;
- `Accept`;
- `Idempotency-Key`;
- `X-Request-ID`;
- условные или иные endpoint-specific headers.

### 6. Request body

Для JSON-body указываются:

- полный пример JSON;
- таблица полей;
- тип каждого поля;
- required/optional;
- nullable/non-nullable;
- допустимые значения и диапазоны;
- формат даты и времени;
- единицы измерения;
- бизнес-смысл поля;
- какие значения backend не принимает на доверии и рассчитывает самостоятельно.

JSON-пример должен быть синтаксически корректным и соответствовать реальному request DTO.

### 7. Успешные ответы

Для каждого возможного успешного status code указываются:

- HTTP status;
- полный пример response body;
- таблица полей;
- семантика nullable и отсутствующих полей;
- headers ответа, если они значимы;
- условия, при которых возвращается именно этот статус.

### 8. Ошибки

Для endpoint-а перечисляются ожидаемые ошибки:

- HTTP status;
- стабильный machine-readable error code;
- безопасное пользовательское сообщение;
- условие возникновения;
- возможность безопасного retry;
- влияние idempotency key;
- пример error response.

Ошибки должны соответствовать централизованному mapper из ADR-0016.

### 9. Идемпотентность

Для изменяющих состояние endpoint-ов указывается:

- требуется ли `Idempotency-Key`;
- scope ключа;
- срок хранения результата;
- поведение при повторе с тем же payload;
- поведение при повторе с другим payload;
- какие поля входят в канонический semantic hash.

### 10. Побочные эффекты и транзакционность

Документ должен кратко указывать видимые внешние последствия:

- какие бизнес-сущности создаются или изменяются;
- проводится ли операция атомарно;
- создаётся ли аудит;
- изменяются ли остатки;
- возникают ли фоновые действия;
- является ли ответ строго актуальным или основанным на eventual-consistent projection.

Внутренние SQL-детали и структура private repository при этом не раскрываются.

### 11. Примеры

Для каждого endpoint-а должны быть представлены как минимум:

- один корректный request;
- один корректный response;
- один типовой error response.

Для сложных endpoint-ов добавляются примеры граничных случаев.

## Общий формат API

`05-api-design.md` должен также централизованно определять общие правила, чтобы не повторять их полностью для каждого endpoint-а:

- base URL и версионирование;
- authentication scheme;
- общий success envelope;
- общий error envelope;
- формат идентификаторов;
- даты и время;
- денежные значения;
- количества и единицы измерения;
- пагинацию;
- сортировку и фильтрацию;
- idempotency;
- request ID и trace correlation;
- правила backward compatibility;
- deprecated endpoint-ы;
- upload/download contracts.

Конкретный endpoint может ссылаться на общий раздел, но не должен скрывать свои специфические требования.

## Статус реализации

Для удобства команды каждый endpoint получает статус:

- `Planned` — контракт спроектирован, реализация отсутствует;
- `In Progress` — реализация ведётся;
- `Implemented` — код и обязательные тесты готовы;
- `Deprecated` — endpoint сохраняется временно, но не должен использоваться новым кодом;
- `Removed` — endpoint больше недоступен, причина и замена зафиксированы в changelog.

Статус `Implemented` разрешён только после проверки соответствия кода и документации.

## Workflow реализации feature

Feature, затрагивающая API, выполняется в следующем порядке:

1. проверить SRS и существующий API design;
2. спроектировать или изменить контракт в `05-api-design.md`;
3. реализовать request/response DTO, handler и use case;
4. использовать централизованный error mapper;
5. добавить contract и integration tests;
6. сверить реальные status codes, headers и JSON с документом;
7. перевести endpoint в статус `Implemented`;
8. включить изменение API-документа в тот же commit или pull request.

Допускается сначала создать `Planned`-контракт, а затем реализовать его. Не допускается объединять feature как завершённую при отсутствии актуальной документации.

## Изменение существующего endpoint-а

При изменении endpoint-а необходимо явно зафиксировать:

- что изменилось;
- является ли изменение backward compatible;
- какие frontend-клиенты затронуты;
- требуется ли новая версия endpoint-а;
- дата deprecation старого контракта;
- миграционный путь.

Переименование, удаление, изменение типа или обязательности поля не выполняется молча.

## Проверка соответствия

Для предотвращения расхождения должны применяться:

- HTTP contract tests;
- JSON golden tests или schema assertions там, где это оправдано;
- проверка status code и response envelope;
- тестирование error mapping;
- code review checklist с обязательным пунктом обновления `05-api-design.md`;
- CI-проверка изменения API-документа для feature, изменяющей маршруты или DTO, когда такая автоматизация будет внедрена.

OpenAPI может быть добавлен как машинно-читаемое дополнение, но не отменяет человекочитаемый `05-api-design.md`. При появлении OpenAPI необходимо определить единый источник генерации, чтобы два описания не расходились.

## Definition of Done для HTTP feature

HTTP feature считается завершённой только если:

1. endpoint реализован в соответствии с SRS;
2. handler не содержит бизнес-логики;
3. request и response DTO отделены от Domain;
4. ошибки проходят через централизованный mapper;
5. authentication и authorization применены корректно;
6. добавлены необходимые тесты;
7. endpoint полностью описан в `docs/05-api-design.md`;
8. примеры JSON соответствуют фактическому контракту;
9. frontend может реализовать интеграцию, не читая backend-код;
10. breaking changes явно задокументированы.

## Consequences

### Positive

- frontend получает единый каталог всех endpoint-ов;
- сокращается зависимость frontend-команды от знания backend-кода;
- API review происходит до или одновременно с реализацией;
- QA может строить проверки по стабильному контракту;
- breaking changes становятся заметными;
- ускоряется onboarding новых разработчиков;
- уменьшается вероятность расхождения DTO и документации.

### Negative

- каждая API-feature требует дополнительного обновления документации;
- без дисциплины документ может устареть;
- ручные JSON-примеры требуют проверки;
- позднее может понадобиться автоматизация синхронизации с OpenAPI.

Эти затраты принимаются, поскольку стоимость рассинхронизации backend и frontend выше стоимости поддержки единого API-каталога.

## Invariants

1. `docs/05-api-design.md` является единым человекочитаемым каталогом HTTP API.
2. Каждый HTTP endpoint описывается в этом файле.
3. API-feature без актуального описания не считается завершённой.
4. URL, method, access rules, request, response и ошибки обязательны для каждого endpoint-а.
5. JSON-примеры соответствуют фактическим DTO.
6. Изменение контракта и документации выполняется в одном change set.
7. Frontend не должен изучать backend-код для восстановления публичного API-контракта.
8. Ошибки соответствуют централизованному error mapper.
9. Breaking changes не выполняются молча.
10. Документация не раскрывает секреты, внутренние stack traces и private implementation details.
