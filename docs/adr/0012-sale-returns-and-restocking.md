# ADR-0012: Возвраты продаж и восстановление складского остатка

- Status: Accepted
- Date: 2026-07-16
- Decision owners: PharmacyCRM team

## Context

Возврат продажи затрагивает одновременно финансовую историю, лимиты возврата и физический остаток. Один `sale_item` может быть списан из нескольких `stock_lots` по FEFO, а один и тот же товар может возвращаться частями несколькими документами.

Связи только с `sale_item` недостаточно: система должна знать, к каким исходным аллокациям относится возвращаемое количество. Иначе невозможно доказать, что товар возвращён в правильный лот и что по конкретной аллокации не возвращено больше, чем было продано.

Кроме того, финансовый возврат не всегда означает возвращение товара в продаваемый остаток. Возможны действия `RESTOCK`, `WRITE_OFF`, `QUARANTINE` и `NO_PHYSICAL_RETURN`.

## Decision

Возврат оформляется отдельным неизменяемым документом `sale_returns` со строками `sale_return_items`.

Для количественной привязки к исходным лотам вводится отдельная сущность:

```text
sale_return_item_allocations
- id
- sale_return_item_id
- sale_item_allocation_id
- returned_quantity_base_units
- created_at
```

Каждая строка возврата распределяется по исходным `sale_item_allocations`. Это распределение является источником истины для проверки лимитов и, при `RESTOCK`, для положительных складских движений.

## Return actions

- `RESTOCK` — товар после проверки возвращается в доступный остаток исходного лота.
- `WRITE_OFF` — физически возвращённый товар не является продаваемым и оформляется отдельной складской операцией списания либо не приходуется вовсе, в зависимости от процесса приёмки.
- `QUARANTINE` — товар физически принят, но не возвращается в продаваемый остаток; для полноценной реализации потребуется отдельный карантинный контур.
- `NO_PHYSICAL_RETURN` — выполняется финансовый возврат без физического прихода товара.

В MVP только `RESTOCK` создаёт положительные движения в доступном остатке. Остальные действия не должны молча увеличивать `stock_lots.quantity_base_units`.

## Transaction order

Частичный возврат проводится в следующем порядке:

1. проверить идемпотентность запроса;
2. начать транзакцию;
3. заблокировать исходную `sales` через `SELECT ... FOR UPDATE`;
4. загрузить и проверить возвращаемые `sale_items`;
5. заблокировать соответствующие `sale_item_allocations` в возрастающем порядке `id`;
6. рассчитать уже возвращённое количество по каждой исходной аллокации, учитывая только проведённые возвраты;
7. проверить, что новое возвращаемое количество не превышает остаток возвратного лимита;
8. для строк с `RESTOCK` собрать связанные `pharmacy_product_id` и заблокировать их в порядке, установленном ADR-0011;
9. заблокировать целевые `stock_lots` в возрастающем порядке `id`;
10. создать `sale_return` и его строки;
11. создать `sale_return_item_allocations`;
12. для `RESTOCK` создать `inventory_operation`, увеличить остатки исходных лотов и добавить положительные `inventory_movements`;
13. обновить агрегированный статус исходной продажи: `PARTIALLY_RETURNED` или `RETURNED`;
14. обновить `inventory_changed_at` только для товаров, чей физический остаток изменился;
15. зафиксировать транзакцию.

## Allocation policy

Возврат должен ссылаться на исходные аллокации. Когда пользователь возвращает часть позиции и не выбирает конкретный лот, backend распределяет возврат детерминированно по исходным аллокациям, например в порядке их `id` или в обратном порядке фактического списания. Выбранное правило должно быть единым и протестированным.

Для MVP принимается порядок `sale_item_allocations.id ASC`, поскольку он стабилен и не зависит от изменяемых атрибутов лота.

## Restocking policy

Возврат в исходный лот допускается только если:

- товар физически принят аптекой;
- лот принадлежит той же аптеке;
- представление товара совпадает с исходной продажей;
- количество подтверждено аптекарем;
- срок годности не истёк;
- состояние упаковки позволяет повторную продажу;
- применимые нормативные правила разрешают возврат препарата в продаваемый оборот.

Если хотя бы одно условие не выполнено, действие `RESTOCK` запрещается. Система должна потребовать `WRITE_OFF`, `QUARANTINE` или `NO_PHYSICAL_RETURN`.

## Alternatives considered

### Возврат только на уровне `sale_item`

Отклонён: одна строка продажи может быть списана из нескольких лотов, поэтому невозможно точно ограничить возврат и восстановить происхождение количества.

### Создание нового лота для каждого возврата

Не используется по умолчанию: это разрывает связь с исходной себестоимостью и заводской партией. Новый лот может понадобиться только для отдельного карантинного или переоценочного процесса, который будет оформлен самостоятельным решением.

### Изменение исходного чека

Запрещено. Продажа остаётся неизменяемой, а возврат является отдельным компенсирующим документом.

## Consequences

### Положительные

- Нельзя вернуть больше, чем было фактически списано из каждого лота.
- Сохраняется полная связь между продажей, лотом, возвратом и движением.
- Частичные и повторные возвраты обрабатываются без изменения исходного чека.
- Финансовый возврат отделён от решения о физическом остатке.

### Отрицательные

- Появляется дополнительная таблица аллокаций возврата.
- Возврат требует нескольких блокировок и более сложных интеграционных тестов.
- Карантин потребует отдельной модели хранения в будущей версии.

## Invariants

1. Возврат всегда ссылается на существующую проведённую продажу.
2. `sale_return_item` ссылается на строку той же продажи.
3. Сумма аллокаций возврата равна `returned_quantity_base_units` строки возврата.
4. Для каждой исходной `sale_item_allocation` суммарно нельзя вернуть больше проданного количества.
5. В лимит учитываются только проведённые возвраты; отменённые черновики его не уменьшают.
6. Только действие `RESTOCK` увеличивает доступный остаток.
7. Каждое увеличение лота имеет положительное immutable-движение в той же транзакции.
8. Исходные `sales`, `sale_items` и `sale_item_allocations` не изменяются.
9. Возврат не может ссылаться на лот другой аптеки.
10. Просроченный товар нельзя вернуть в продаваемый остаток.
11. Сумма финансового возврата рассчитывается backend на основании снимков исходной продажи и правил возврата.
12. Повтор запроса с тем же idempotency key не создаёт второй возврат.

## Schema implications

`database-design.md` должен быть дополнен таблицей `sale_return_item_allocations` и ограничениями/индексами для проверки возвратов:

```sql
CREATE TABLE sale_return_item_allocations (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    sale_return_item_id uuid NOT NULL
        REFERENCES sale_return_items(id) ON DELETE RESTRICT,
    sale_item_allocation_id uuid NOT NULL
        REFERENCES sale_item_allocations(id) ON DELETE RESTRICT,
    returned_quantity_base_units bigint NOT NULL
        CHECK (returned_quantity_base_units > 0),
    created_at timestamptz NOT NULL DEFAULT now(),
    UNIQUE (sale_return_item_id, sale_item_allocation_id)
);

CREATE INDEX idx_return_allocations_source
ON sale_return_item_allocations (sale_item_allocation_id);
```

Проверка совокупного лимита возврата является транзакционным инвариантом и не выражается обычным построчным `CHECK`.

## Testing requirements

Обязательны интеграционные тесты:

- частичный возврат одной позиции;
- несколько последовательных частичных возвратов;
- два конкурентных возврата одной продажи;
- попытка вернуть больше проданного;
- возврат позиции, списанной из нескольких лотов;
- финансовый возврат без изменения склада;
- запрет restock просроченного товара;
- идемпотентный повтор возврата;
- полный rollback при ошибке создания движения.
