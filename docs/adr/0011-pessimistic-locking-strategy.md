# ADR-0011: Детерминированная стратегия пессимистических блокировок

- Status: Accepted
- Date: 2026-07-16
- Decision owners: PharmacyCRM team

## Context

При параллельных продажах одного и того же товара несколько транзакций могут одновременно видеть одинаковый доступный остаток и пытаться списать его из одних и тех же складских лотов по FEFO.

Блокировка только выбранных `stock_lots` усложняет реализацию: транзакции могут выбрать пересекающиеся наборы лотов и захватить их в разном порядке, что повышает риск взаимных блокировок.

При этом один чек может содержать несколько товаров. Поэтому блокировка одной строки `pharmacy_products` защищает только продажу одного товара, но сама по себе не исключает deadlock для многострочного чека, если разные транзакции блокируют товары в разном порядке.

## Decision

Для MVP используется пессимистическая сериализация продаж на уровне `pharmacy_products`.

Перед FEFO-аллокацией транзакция обязана:

1. собрать уникальные `pharmacy_product_id` всех позиций чека;
2. отсортировать их по UUID в возрастающем порядке;
3. захватить строки `pharmacy_products` в этом же порядке через `SELECT ... FOR UPDATE`;
4. только после захвата всех родительских блокировок читать доступные `stock_lots` и рассчитывать FEFO-аллокации;
5. обновлять задействованные `stock_lots` в детерминированном порядке: `expiration_date`, `received_at`, `id`;
6. завершать все изменения в одной транзакции PostgreSQL с уровнем изоляции `READ COMMITTED`.

Пример блокировки:

```sql
SELECT id
FROM pharmacy_products
WHERE id = ANY($1::uuid[])
ORDER BY id
FOR UPDATE;
```

После блокировки выполняется FEFO-запрос:

```sql
SELECT id,
       quantity_base_units,
       expiration_date,
       received_at
FROM stock_lots
WHERE pharmacy_product_id = $1
  AND status = 'ACTIVE'
  AND quantity_base_units > 0
  AND expiration_date >= CURRENT_DATE
ORDER BY expiration_date, received_at, id;
```

## Transaction order

Продажа проводится в следующем порядке:

1. проверить или зарегистрировать идемпотентный запрос;
2. начать транзакцию;
3. заблокировать все `pharmacy_products` в отсортированном порядке;
4. повторно загрузить актуальные цены, правила продажи и доступные лоты;
5. проверить рецептурные ограничения и вычислить количество базовых единиц;
6. рассчитать FEFO-аллокации;
7. проверить достаточность суммарного остатка;
8. создать `inventory_operation` и документ продажи;
9. создать `sale_items` и `sale_item_allocations`;
10. атомарно уменьшить остатки лотов;
11. создать отрицательные `inventory_movements`;
12. обновить `pharmacy_products.inventory_changed_at`;
13. зафиксировать транзакцию.

Создание документов и движений должно учитывать фактические внешние ключи DDL. Порядок вставки выбирается так, чтобы все ссылки были валидны внутри одной транзакции.

## Alternatives considered

### Блокировка только `stock_lots`

Отклонена для MVP из-за более сложной повторной выборки, необходимости строгого порядка блокировок и повышенного риска ошибок при параллельной FEFO-аллокации.

### Advisory locks

Отклонены как основная стратегия. Они не выражают связь через обычные строки и внешние ключи, требуют собственного пространства ключей и усложняют диагностику.

### Уровень изоляции SERIALIZABLE

Отклонён как основной механизм. Он не устраняет необходимость корректной обработки конфликтов и потребует обязательных повторов транзакций. Для MVP явные блокировки проще контролировать и тестировать.

## Consequences

### Положительные

- FEFO-аллокация одного товара в одной аптеке выполняется последовательно.
- Исключается одновременное списание одного и того же остатка без проверки.
- Единый порядок блокировок существенно снижает риск deadlock для многострочных чеков.
- Алгоритм легко воспроизводится в интеграционных тестах.

### Отрицательные

- Параллельные продажи одного товара в одной аптеке временно выстраиваются в очередь.
- Долгие транзакции могут задерживать следующие продажи этого товара.
- Код обязан соблюдать единый порядок блокировок во всех сценариях, включая возвраты, списания и корректировки.

## Invariants

1. Все `pharmacy_product_id` одной транзакции блокируются до чтения лотов.
2. Все родительские строки блокируются только в возрастающем порядке UUID.
3. После захвата блокировок цены, правила продажи и остатки перечитываются из БД.
4. FEFO учитывает только активные, непросроченные лоты с положительным остатком.
5. Сумма `sale_item_allocations.quantity_base_units` равна `sale_item.quantity_base_units`.
6. Каждое уменьшение `stock_lots.quantity_base_units` имеет соответствующее отрицательное движение.
7. Ни один лот не может получить отрицательный остаток.
8. Frontend не задаёт итоговую цену и фактические лоты списания.
9. Транзакции должны быть короткими: внешние API, сеть и тяжёлые вычисления внутри блокировки запрещены.
10. Ошибки PostgreSQL deadlock/serialization должны распознаваться и безопасно повторяться ограниченное число раз, даже при детерминированном порядке блокировок.

## Testing requirements

Необходимы интеграционные тесты PostgreSQL:

- две одновременные продажи одного товара при достаточном остатке;
- две одновременные продажи при недостаточном суммарном остатке;
- два многострочных чека с одинаковыми товарами в разном входном порядке;
- списание одной позиции из нескольких лотов;
- идемпотентный повтор успешно завершённой продажи;
- отсутствие частично созданных документов после rollback.
